1. 冲突是怎么产生的？—— 分布式系统的日常

📶 网络分区
设备A改了数据但没来得及同步到云端，设备B也改了同一条数据。
📱 离线修改
手机离线时修改了资产，平板在线修改了同一资产，联网后冲突。
⚡ 并发操作
几乎同一时刻两个设备提交修改，服务器无法判断先后。

2. 主流冲突解决策略（通俗版）

策略名称原理适合场景
⚖️ 最后写入获胜 (LWW)谁的时间戳最新，谁覆盖对方简单数据，允许一定覆盖
🤝 手动合并弹出窗口让用户选择保留哪个版本重要数据（金额、审核状态）
🧩 自动合并（字段级）比较每个字段，各自保留最新修改复杂对象，如资产不同字段分别修改
📌 版本向量+增量同步记录每次修改的版本，只同步变化部分高效协同，分布式数据库常用

🗂️ 鸿蒙分布式数据库内置了“冲突解决回调”，我们只要实现一个合并函数即可！

3. 鸿蒙中如何实现？—— DistributedDataKit 冲突解决API

// 伪代码流程（具体API以官方最新为准，但思路通用） // 1. 创建分布式数据库时，设置冲突解决策略 // 2. 实现 resolveConflict 方法 // 3. 比较本地和远端的版本号/时间戳，返回合并后的数据 // 示例思路：最后写入获胜 + 字段级合并 resolveConflict(localEntry, remoteEntry) { if (localEntry.timestamp > remoteEntry.timestamp) { return localEntry; // 本地更新，保留本地 } else if (remoteEntry.timestamp > localEntry.timestamp) { return remoteEntry; // 远端更新，覆盖本地 } else { // 相同时间戳，则人工合并关键字段（比如价格取平均？抛给用户？） return manualMerge(localEntry, remoteEntry); } }

⚙️ 技能实训 · 动手解决“资产价格冲突”

✅ 实训目标：在模拟分布式环境下，编写冲突处理函数，实现最后写入获胜和字段级合并

📁 实训准备：搭建模拟冲突环境

  • 打开已有的HarmonyOS项目（资产管理系统），确保已集成分布式数据库（参考之前课程）。
  • 在数据库中插入一条测试资产：id=1001, name=MacBook, price=8000, timestamp=当前时间。
  • 模拟冲突数据： 手动构造远端版本（例如修改price为8500，时间戳+1）和本地版本（price为8200，时间戳稍旧）。

    • 🧪 实训任务一：实现“最后写入获胜”策略（20分钟）

  • 在分布式数据库的帮助类中，实现一个冲突处理方法 resolveLWW(local, remote)。
  • 比较 timestamp 字段（数值越大表示越新），返回较新的那条。
  • 编写测试代码，分别模拟两个版本，调用方法并打印合并后的价格。
  • 验证： 老师提供两组timestamp值，学生运行代码后给出正确结果。
    • // 代码示例（ArkTS） function resolveLWW(local: AssetEntry, remote: AssetEntry): AssetEntry { // 最后写入获胜: 比较时间戳 if (local.timestamp > remote.timestamp) { console.info('保留本地最新数据', local.price); return local; } else if (remote.timestamp > local.timestamp) { console.info('采用远端数据', remote.price); return remote; } else { // 时间戳相等，可以保留本地，也可根据业务决定 console.warn('时间戳相同，默认保留本地'); return local; } } // 测试代码 let local = { id: '1001', price: 8200, timestamp: 1715400000 }; let remote = { id: '1001', price: 8500, timestamp: 1715403600 }; let result = resolveLWW(local, remote); console.log(`最终价格: ${result.price}`); // 应该输出 8500

    🧪 实训任务二：升级为字段级合并（更智能）

  • 需求： 如果本地改了name字段，远端改了price字段，它们应该合并，而不是互相覆盖。
  • 新建方法 mergeByField(local, remote)，比较每个字段的最后修改时间。
  • 为每个字段添加一个 fieldTimestamp 记录（例如 lastModified_name, lastModified_price）。
  • 合并时分别取每个字段较新的值，组装成新对象返回。
  • 打印合并前后的数据，验证合并正确性。
    • // 字段级合并示例（简化） interface AssetEntry { id: string; name: string; price: number; ts_name: number; // name字段修改时间 ts_price: number; // price字段修改时间 } function mergeByField(local: AssetEntry, remote: AssetEntry): AssetEntry { let merged: AssetEntry = { ...local }; // 复制本地副本 // name字段取最新的 if (remote.ts_name > local.ts_name) { merged.name = remote.name; merged.ts_name = remote.ts_name; } // price字段取最新的 if (remote.ts_price > local.ts_price) { merged.price = remote.price; merged.ts_price = remote.ts_price; } console.log('字段级合并完成'); return merged; }

    ⭐ 提示：实际鸿蒙分布式数据库的冲突回调会提供两个数据快照，你只需要返回合并后的结果。字段级合并更精细，适合资产管理系统！

    🧪 实训任务三：集成到分布式数据库回调中（25分钟）

  • 查阅鸿蒙文档，找到 DistributedObject 或 KVManager 的冲突监听接口（如 onConflict）。
  • 将刚才写的 mergeByField 作为冲突解决函数注册到数据库配置中。
  • 实际使用两台真机或模拟器同步数据，修改同一条记录的不同字段，观察最终结果是否“合并成功”。
  • 添加日志输出，显示冲突发生时调用了我们的合并逻辑，并打印合并细节。
  • 小组展示： 每组选一名代表演示“多设备修改资产不同字段，最后数据完好合并”的效果。
