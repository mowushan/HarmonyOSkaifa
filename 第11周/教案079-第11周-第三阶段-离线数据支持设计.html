🧩 知识讲解 · 离线数据支持

1. 为什么要做离线支持？—— 真实场景 + 大赛加分

📱 真实故事： 实训车间信号不好，学生想录入设备资产，结果应用白屏崩溃……评委看到会直接扣分！

离线设计的三大好处：

  • ✅ 用户体验好：没网也能看历史数据、录入操作不丢失。
  • ✅ 应用更健壮：不会因为网络请求失败而闪退。
  • ✅ 大赛作品亮点：体现工程化思考，比单纯调API的学生高一个层次。

📌 核心原则： 网络请求成功 → 存到本地；网络请求失败 → 读本地数据 + 暂存操作（待同步）。

2. 网络状态检测 —— 应用的“眼睛”

鸿蒙提供了 @ohos.net.connection 模块，可以检测当前网络类型（Wi-Fi/蜂窝/无网络），并监听网络变化。

// 导入网络连接模块 import connection from '@ohos.net.connection'; // 检测当前网络状态 function checkNetworkStatus() { let netHandle = connection.getDefaultNetSync(); if (netHandle) { let capabilities = connection.getNetCapabilitiesSync(netHandle); if (capabilities && capabilities.linkUpBandwidthKbps > 0) { console.info("网络已连接"); return true; } } console.info("无网络连接"); return false; }

3. 离线数据缓存策略 —— 旁路缓存(Cache-Aside)模式

📖 读取流程

1️⃣ 先查本地数据库/Preferences
2️⃣ 如果有数据，直接展示（离线模式）
3️⃣ 如果没有 or 需要最新 + 有网 → 发起网络请求
4️⃣ 网络请求成功后，更新本地缓存 + 刷新UI

✍️ 写入流程

1️⃣ 立即更新本地数据库（用户无感知）
2️⃣ 如果在线，立即同步到云端
3️⃣ 如果离线，将请求存入“待同步队列”，等网络恢复后再同步

💡 我们不需要实现复杂的同步协议，今天只做核心功能：读数据时“优先读本地”，写数据时“本地先存+标记待同步” 就足够惊艳评委了！

🧪 实训任务一：添加网络状态检测工具类（15分钟）

  • 在 common/utils 目录下新建 NetworkUtil.ets。
  • 粘贴以下代码（手敲一行或老师共享），理解每一部分作用。
    • // NetworkUtil.ets import connection from '@ohos.net.connection'; export class NetworkUtil { static isNetworkAvailable(): boolean { try { let netHandle = connection.getDefaultNetSync(); if (!netHandle) { return false; } let capabilities = connection.getNetCapabilitiesSync(netHandle); return capabilities && capabilities.linkUpBandwidthKbps > 0; } catch (err) { console.error("网络检测异常:" + JSON.stringify(err)); return false; } } // 监听网络变化（可选，用于实时同步） static registerNetworkCallback(callback: (isOnline: boolean) => void) { let netSpec: connection.NetSpecifier = { capabilities: [connection.NetCap.NET_CAPABILITY_INTERNET] }; connection.registerNetReceiver(netSpec, { onNetAvailable: () => callback(true), onNetLost: () => callback(false) }); } }

    🧪 实训任务二：改造数据加载方法 —— 离线优先（25分钟）

    核心逻辑： loadAssetList() 中先查本地数据库，有网时再从网络拉取最新数据并更新本地。

    // AssetList.ets 片段 import { NetworkUtil } from '../common/utils/NetworkUtil'; import { DatabaseHelper } from '../database/DatabaseHelper'; @Entry @Component struct AssetList { @State assetList: AssetItem[] = []; async aboutToAppear() { // 1. 先展示本地数据（离线兜底） await this.loadLocalAssets(); // 2. 如果有网，拉取远程数据并合并 if (NetworkUtil.isNetworkAvailable()) { await this.syncRemoteAssets(); } else { prompt.showToast({ message: "当前无网络，展示本地缓存数据" }); } } async loadLocalAssets() { let localData = await DatabaseHelper.getAllAssets(); if (localData.length > 0) { this.assetList = localData; } } async syncRemoteAssets() { // 调用网络接口获取最新资产列表 let remoteList = await ApiService.fetchAssets(); if (remoteList) { // 存到本地数据库 await DatabaseHelper.saveAssets(remoteList); // 刷新UI this.assetList = remoteList; prompt.showToast({ message: "已更新云端最新数据" }); } } }

    🔧 教师演示 + 学生分步实现：

    1. 先让学生写出 loadLocalAssets 读取本地数据库的方法（上周已经实现过）。
    2. 再添加网络检测，分别测试“有网模式”和“飞行模式”下界面显示差异。
    3. 重点强调：离线时展示的数据不能是空的，否则应用就废了。

    🧪 实训任务三：断网写入-请求暂存队列（拓展实战，20分钟）

    场景：学生在离线时新增了一条资产，如何保证不丢失并在网络恢复后自动提交？

  • 新建一个表 pending_operations 存储待同步的请求（方法名、参数、时间戳）。
  • 新增资产时先写入本地数据库，同时插入一条待同步记录。
  • 监听网络恢复事件，自动执行队列中的所有请求。
    • // 新增资产方法（离线安全版） async addAsset(newAsset: AssetItem) { // 1. 立即插入本地数据库（用户马上看到） await DatabaseHelper.insertAsset(newAsset); this.assetList = [newAsset, ...this.assetList]; // 2. 保存请求到待同步队列 let pendingOp = { url: "/api/assets", method: "POST", body: JSON.stringify(newAsset), timestamp: Date.now() }; await DatabaseHelper.savePendingOp(pendingOp); // 3. 如果有网，立即同步；否则等待网络恢复 if (NetworkUtil.isNetworkAvailable()) { await this.flushPendingQueue(); } } async flushPendingQueue() { let queue = await DatabaseHelper.getPendingOps(); for (let op of queue) { // 执行网络请求 let success = await ApiService.dispatch(op); if (success) { await DatabaseHelper.deletePendingOp(op.id); } } }
