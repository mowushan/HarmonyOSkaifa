🧩 知识讲解 · 数据同步设计思想

1. 为什么需要数据同步？

📱+☁️ 我们的资产管理App既要支持离线使用（本地SQLite），又要支持多设备登录（手机+平板）。
核心问题： 在手机上新加一笔资产，平板上怎样立刻看到？答案就是数据同步。

  • 全量同步：每次拉取所有数据（简单但流量大）。
  • 增量同步：只拉取上次同步之后变化的数据（高效）。
  • 同步方向：本地→云端（上传），云端→本地（拉取）。

2. 同步架构设计（简洁版）

📦 本地数据层

SQLite数据库 + 一个“同步标记”字段（sync_status: 0未同步, 1已同步）

🌐 云端API

GET /assets 获取所有资产
POST /assets 新增资产
PUT /assets/{id} 更新资产
DELETE /assets/{id} 删除资产

🔄 同步管理器

1. 上传本地未同步的数据 → 2. 拉取云端最新数据 → 3. 合并到本地数据库

3. 冲突处理策略（小白友好版）

场景： 手机端修改了资产名称，同时平板端也修改了同一资产，谁为准？
我们的策略： “最后一次修改时间戳”大的获胜。云端和本地每条记录都存一个 last_modified 时间戳，同步时比较时间戳，谁新用谁。

// 数据结构示例（简化） interface Asset { id: number; name: string; amount: number; last_modified: number; // 毫秒时间戳 sync_status: number; // 0=待上传, 1=已同步 }

  • 若没有，执行ALTER TABLE语句添加字段。老师会提供迁移脚本。
  • 在应用启动时，检查本地数据库版本号，执行升级。

    🔧 实训二：编写 SyncManager 核心类

    // SyncManager.ets 核心代码 import http from '@ohos.net.http'; import relationalStore from '@ohos.data.relationalStore'; export class SyncManager { private baseUrl = 'https://your-mock-api.com/assets'; // 替换为真实API private store: relationalStore.RdbStore; constructor(store: relationalStore.RdbStore) { this.store = store; } // 1. 上传本地所有未同步的数据 async uploadLocalChanges(): Promise<void> { let unsynced = await this.queryUnsyncedAssets(); for (let asset of unsynced) { await this.uploadSingleAsset(asset); } } private async queryUnsyncedAssets(): Promise<any[]> { let predicates = new relationalStore.RdbPredicates('assets'); predicates.equalTo('sync_status', 0); let resultSet = await this.store.query(predicates); let list = []; while (resultSet.goToNextRow()) { list.push({ id: resultSet.getLong(resultSet.getColumnIndex('id')), name: resultSet.getString(resultSet.getColumnIndex('name')), amount: resultSet.getDouble(resultSet.getColumnIndex('amount')), last_modified: resultSet.getLong(resultSet.getColumnIndex('last_modified')) }); } resultSet.close(); return list; } private async uploadSingleAsset(asset: any): Promise<void> { let httpRequest = http.createHttp(); let response = await httpRequest.request(this.baseUrl, { method: http.RequestMethod.POST, header: { 'Content-Type': 'application/json' }, extraData: JSON.stringify(asset) }); if (response.responseCode === 200) { // 更新本地同步状态 let updatePredicates = new relationalStore.RdbPredicates('assets'); updatePredicates.equalTo('id', asset.id); await this.store.update(updatePredicates, { sync_status: 1 }); } httpRequest.destroy(); } // 2. 从云端拉取所有资产并合并到本地（简单全量合并） async pullAndMerge(): Promise<void> { let httpRequest = http.createHttp(); let response = await httpRequest.request(this.baseUrl, { method: http.RequestMethod.GET }); let cloudAssets = JSON.parse(response.result as string); for (let cloudItem of cloudAssets) { let local = await this.findLocalById(cloudItem.id); if (!local) { // 本地没有，直接插入 await this.insertAsset(cloudItem); } else { // 比较时间戳，云端更新则覆盖 if (cloudItem.last_modified > local.last_modified) { await this.updateAsset(cloudItem); } } } httpRequest.destroy(); } // 辅助方法略（findLocalById, insertAsset, updateAsset） }

    📢 教师讲解：重点解释 sync_status 的作用——只有新增或修改过的记录才需要上传，避免重复请求。

    🔧 实训四：冲突模拟与解决

    小组挑战： 两个人分别用不同设备（或同一个设备切换账号）修改同一条资产，观察同步后哪一方覆盖哪一方。并思考为什么需要时间戳方案。

    教师演示：故意让手机和平板同时修改，然后分别点击同步，最后查看数据以最后修改时间为准。

    🔧 实训五：错误处理与重试机制

  • 当网络请求失败时，不要丢弃未同步的数据，保留 sync_status=0。
  • 增加“手动重试”按钮，或使用后台定时任务重试。
  • 提示用户“当前有未同步数据，请连接网络后同步”。
