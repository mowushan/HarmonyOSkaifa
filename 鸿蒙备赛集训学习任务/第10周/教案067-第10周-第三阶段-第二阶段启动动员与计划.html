2. 鸿蒙数据存储三兄弟 —— 怎么选？

🥇 Preferences

键值对存储，适合少量配置（如用户主题、是否首次启动）。
例子： 记住用户是否登录过。

🥈 关系型数据库 (RelationalStore)

支持SQL，适合资产清单这种结构化数据，可查询、排序、关联。
例子： 资产管理表（id, 名称, 金额）。

🥉 云端 / 网络

需要联网，可实现多设备同步。
例子： 从华为云获取资产备份。

✅ 我们的选择： 资产核心数据用关系型数据库，用户设置用Preferences，后续第三阶段末期对接云端API。

3. 事务（Transaction） —— 要么全成功，要么全不成功

💰 生活案例： 银行转账——从A账户扣100元，B账户加100元。如果扣完钱后系统崩溃，但B没到账，就出大问题！事务保证两个操作一起完成或一起回滚。

ACID 四个特性（简单记）：

  • 原子性： 不可分割。扣款+加款是一个整体。
  • 一致性： 转账前后总金额不变。
  • 隔离性： 多个人同时转账互不干扰。
  • 持久性： 成功后数据永久保存。

💡 下节课我们会用代码演示“批量删除资产时如何用事务保证数据安全”。

    • 🔧 实训四（合作探究）：事务模拟小剧场

    🎭 角色扮演游戏（10分钟）：

    • 场景： 批量删除3个资产。
    • 角色A： 数据库操作员（执行删除）
    • 角色B： 事务管理员（控制begin/commit/rollback）
    • 角色C： 观察员（记录成功/失败情况）

    玩法： 在白板或便签上写下删除步骤。假设删除到第二个资产时网络异常，事务管理员执行 rollback，所有删除操作撤销。让学生口头说明“如果没有事务，数据会怎样”。

    🎯 目的：下节课写SQLite事务代码时，脑子里已经清楚流程。

    // 下节课会学到的伪代码示意 let store = await relationalStore.getStore(context, config); await store.beginTransaction(); // 开启事务 try { await store.delete(predicates1); await store.delete(predicates2); await store.delete(predicates3); await store.commit(); // 全部成功 -> 提交 } catch(e) { await store.rollback(); // 出错 -> 回滚 }
