🧩 知识讲解 · UI测试与调试工具箱

1. UI测试——给界面做“体检”

🖥️ 什么是UI测试？ 就是手动或自动检查每个界面元素是否正常工作。

为什么要做？ 防止“用户点按钮没反应”、“列表显示不全”、“跳转闪退”等尴尬问题。

测试清单（我们手动做）：

  • ✅ 每个按钮/文本输入框是否可见、可点
  • ✅ 页面跳转是否携带正确数据
  • ✅ 列表滚动是否流畅
  • ✅ 不同屏幕尺寸下布局是否错位

2. DevEco Studio 调试神器——断点+监视

🔴 断点（Breakpoint）

在代码行号旁单击，程序运行到此处会“暂停”，让你观察当前状态。

▶️ 单步执行（Step Over）

一行一行执行代码，追踪执行路径。

👁️ 变量监视（Watch）

查看变量当前的值，检查是否符合预期。

3. 日志调试——最直接的“侦查手段”

// ArkTS中打印日志（类似Python的print） console.log("按钮被点击了！当前计数：" + count); // 鸿蒙官方推荐 hilog（更强大） import { hilog } from '@ohos.hilog'; hilog.info(0x0000, 'MyApp', '用户点击了登录按钮');

小技巧： 日志就像“探照灯”，在关键位置打印变量值，能帮你快速找到哪里出错了。

⚙️ 技能实训 · 实战演练：调试资产录入页面

✅ 本次实训目标：学会打断点、看日志、修复UI常见bug

🔧 实训准备：打开上阶段的“资产录入”项目

  • 在DevEco Studio中打开你的 AssetManagement 项目（如果没有，老师会提供基础模板）。
  • 确保可以运行在模拟器或真机上。
  • 找到 pages/AddAsset.ets 文件（资产录入页）。

    • 🧪 实训任务一：手动UI测试 · 找茬大挑战（15分钟）

  • 测试清单： 运行App，进入“资产录入”页面，依次测试：输入框能否输入、提交按钮是否响应、返回按钮是否正常。
  • 故意制造“bug”体验： 老师提供一个有bug版本（比如按钮无反应、输入框无法聚焦），分组找出所有问题，记录在“bug清单”中。
  • 汇报展示： 每组派代表说明发现了哪些问题，老师总结常见UI问题类型。

    • 🐞 常见UI Bug示例：

    • 按钮绑定了错误的事件方法
    • 输入框没有绑定 @State 变量
    • 页面跳转时路由路径写错
    • 布局未设置宽高导致组件消失

    🧪 实训任务二：日志调试 · 给代码装“摄像头”（20分钟）

  • 在提交按钮的点击事件中添加日志： 在 AddAsset.ets 中找到 Button 的 onClick，添加如下代码：
    Button('提交资产') .onClick(() => { // 添加日志，观察是否执行到这里 console.log("提交按钮被点击了"); console.log("资产名称：" + this.assetName); // ... 原有保存逻辑 })
  • 运行并点击按钮，查看“Log”面板（底部Console）： 观察日志是否输出，变量值是否正确。
  • 故意制造错误： 注释掉变量声明，看控制台报错信息，学会读“红色错误提示”。
  • 小组任务： 在页面加载时（aboutToAppear）也添加日志，验证生命周期是否执行。

    • 📢 调试口诀：“日志加得妙，bug跑不掉；变量打印全，问题一眼见！”

    🧪 实训任务三：断点调试 · 让程序“暂停”看病（25分钟）

  • 设置断点： 在 onClick 回调的第一行代码左边单击，出现红色圆点。
  • 以调试模式运行： 点击虫子图标 “Debug” 按钮，程序启动后会停在断点处。
  • 单步执行： 按 F8 逐行执行，观察变量面板中 this.assetName 的值变化。
  • 修改变量值： 在调试过程中，可以右键变量选择 “Set Value”，临时修改，测试不同分支逻辑。
  • 分组挑战： 给“表单验证”函数添加断点，观察当输入为空时，验证逻辑是否正确触发。
    • // 示例：表单验证函数 validateForm(): boolean { if (this.assetName === '') { console.log("资产名称为空，无法提交"); return false; } return true; }

    ✅ 调试成功标志： 能在断点处看到变量，并能单步执行，观察到执行路径。

    🧪 实训任务四：修复一个真实Bug · 界面不刷新问题（15分钟）

  • 场景： 提交资产后，列表页没有立即显示新数据。这是典型“数据没刷新”问题。
  • 定位方法： 在列表页的 aboutToAppear() 和数据加载函数中添加日志，确认是否重新加载。
  • 解决方案： 使用状态管理（@State）或事件通知（emitter）触发列表刷新。老师演示一种简单修复：提交后重新查询数据库并更新列表状态。
  • 动手修复： 每组尝试修改代码，让提交后列表自动显示新资产。成功后老师验收。

    • 🏆 终极挑战： 自己制造一个界面Bug（如按钮点一次后消失），然后利用调试工具找到原因并修复。看谁最快！
