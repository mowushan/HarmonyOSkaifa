🧩 知识讲解 · 表单验证的艺术

1. 为什么需要表单验证？—— 别让“脏数据”污染数据库

🚫 糟糕的用户体验场景：

  • ❌ 用户没填资产名称，点了保存，系统直接崩溃……
  • ❌ 手机号填了“123”，居然保存成功了！
  • ❌ 资产价格填了“abc”，后台报错，用户一脸懵。

✅ 有了表单验证： 在用户点“保存”之前，我们就在前端把错误拦住，并告诉用户“哪里错了、怎么改”。

🎯 前端验证的作用： 提升体验 + 减少无效请求。真正的安全还需要后端验证，但前端是给用户的第一道友好提醒。

2. 常见验证规则（以资产录入为例）

字段验证规则错误提示示例
资产名称必填，2-30个字符“资产名称不能为空” / “长度需2-30字”
资产编号必填，格式：ASSET-数字，如 ASSET-001“编号格式不正确，示例：ASSET-001”
资产价格必填，正数，最多两位小数“请输入有效价格（如 1999.99）”
负责人电话11位手机号（1开头）“请输入正确的11位手机号”
购置日期必填，日期格式 yyyy-mm-dd“请选择购置日期”

3. 正则表达式速成（够用就行）

📱 手机号

^1[3-9]\d{9}$

1开头，第二位3-9，后面9位数字

💰 金额（两位小数）

^\d+(\.\d{1,2})?$

整数或小数，最多两位小数

🏷️ 资产编号自定义

^ASSET-\d{3}$

ASSET- + 三位数字

🔧 小技巧：ArkTS里用 let reg = /^1[3-9]\d{9}$/; reg.test(phone) 来验证。

3. 我们将基于已有代码添加验证逻辑。

🧪 实训任务一：添加状态变量存储错误信息（10分钟）

  • 在组件内定义验证错误消息变量：
    @State nameError: string = '' @State codeError: string = '' @State priceError: string = '' @State phoneError: string = '' @State dateError: string = ''
  • 在输入框下方添加错误提示文本组件：
    Text(this.nameError) .fontSize(12) .fontColor('#ff4d4f') .margin({ top: 4 }) .visibility(this.nameError === '' ? Visibility.None : Visibility.Visible)
  • 同理为每个字段添加对应的错误文本。 注意 visibility 控制显示隐藏。

    • 🧪 实训任务二：编写验证函数（核心！）

  • 定义一个验证方法 validateForm()，返回 boolean：
    // 验证资产名称 if (!this.assetName || this.assetName.trim().length === 0) { this.nameError = '资产名称不能为空' isValid = false } else if (this.assetName.length < 2 || this.assetName.length > 30) { this.nameError = '资产名称长度需2-30个字符' isValid = false } else { this.nameError = '' } // 验证资产编号格式 ASSET-XXX const assetCodeReg = /^ASSET-\d{3}$/ if (!this.assetCode) { this.codeError = '资产编号不能为空' isValid = false } else if (!assetCodeReg.test(this.assetCode)) { this.codeError = '编号格式错误，示例：ASSET-001' isValid = false } else { this.codeError = '' } // 验证价格（数字，最多两位小数） const priceReg = /^\d+(\.\d{1,2})?$/ if (!this.price) { this.priceError = '价格不能为空' isValid = false } else if (!priceReg.test(this.price) || parseFloat(this.price) <= 0) { this.priceError = '请输入有效价格（正数，最多两位小数）' isValid = false } else { this.priceError = '' } // 验证手机号 const phoneReg = /^1[3-9]\d{9}$/ if (!this.phone) { this.phoneError = '联系电话不能为空' isValid = false } else if (!phoneReg.test(this.phone)) { this.phoneError = '请输入正确的11位手机号' isValid = false } else { this.phoneError = '' } // 验证日期 if (!this.purchaseDate) { this.dateError = '请选择购置日期' isValid = false } else { this.dateError = '' } return isValid

    • 🧪 实训任务三：实时验证（输入时清除错误）

  • 为每个输入框添加 onChange 事件：
    TextInput({ text: $$this.assetName }) .onChange(() => { // 输入时立即清除该字段的错误 if (this.nameError !== '') { this.nameError = '' } })
  • 对 price 和 phone 同理添加 onChange 清错。
  • 解释： 这样用户在修正时，红字会消失，体验更丝滑。

    • 🧪 实训任务四：集成提交验证（完整流程）

  • 修改“保存”按钮的点击事件：
    Button('保存资产') .onClick(() => { if (this.validateForm()) { // 验证通过，执行保存逻辑（后续会对接数据库） prompt.showToast({ message: '资产信息验证通过！' }) // 这里可以调用保存数据的方法 } else { prompt.showToast({ message: '请根据提示修正表单错误' }) } })
  • 测试： 故意留空或输入错误格式，观察错误提示和提交拦截效果。

    • 🧪 实训任务五（挑战）：增加“失焦验证”

    在输入框失去焦点（onBlur）时单独验证该字段，实时提醒。

    TextInput({ text: $$this.assetCode }) .onBlur(() => { // 单独验证资产编号 const codeReg = /^ASSET-\d{3}$/ if (this.assetCode && !codeReg.test(this.assetCode)) { this.codeError = '编号格式错误，示例：ASSET-001' } else if (!this.assetCode) { this.codeError = '资产编号不能为空' } else { this.codeError = '' } })

    💡 这样用户在填完一项后立刻知道是否正确，避免最后集中报错造成焦虑。

    🎮 互动活动 · 验证攻防擂台

    「找茬小分队」vs「验证卫士」

    规则： 每组轮流派一人上台演示自己的资产录入页，另一组尝试“破坏”（故意输入各种奇葩数据，比如空值、特殊符号、超长文本）。看谁能成功拦截所有非法输入！

    1. 第一轮： 必填项攻击（留空提交）→ 必须提示“不能为空”
    2. 第二轮： 格式攻击（错误的手机号、价格字母、编号乱写）→ 必须给出具体格式错误提示
    3. 第三轮： 边界攻击（价格负数、价格超长小数、手机号12位）→ 验证要严格

    🏆 奖励： 守擂成功组（拦截率100%）获得“验证卫士”称号 + 平时分加分。攻擂方若找到漏洞，获得“金牌测试员”奖。

    ✨ 通过这个游戏，大家能发现验证逻辑的漏洞，进一步优化代码。
