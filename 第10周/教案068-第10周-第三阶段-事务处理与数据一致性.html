🧩 知识讲解 · 事务的魔法世界

1. 为什么需要事务？—— 一个惨痛的教训

🏦 场景模拟： 资产管理系统里，你要把一个资产从“库房A”转移到“库房B”。

步骤1：从库房A库存减1
步骤2：向库房B库存加1

👿 如果步骤1执行成功，步骤2断电/崩溃了怎么办？ 数据就乱套了——资产凭空消失！

✅ 事务就是用来解决这个问题的：把两个步骤打包成一个“原子操作”，要么都做，要么都不做。

2. 事务的四大特性（ACID）

特性解释生活类比
原子性(Atomicity)事务中的操作要么全部成功，要么全部失败吃套餐：汉堡+可乐+薯条，要么全吃，要么全退
一致性(Consistency)事务前后数据保持逻辑正确转账前后总金额不变
隔离性(Isolation)多个事务互不干扰两个人同时改同一数据，不会互相覆盖
持久性(Durability)事务一旦提交，数据永久保存存钱后即使停电，钱也不会消失

3. 鸿蒙中的事务API

// 获取数据库实例（之前已学过）
let store = await relationalStore.getRdbStore(context, config);

// 开启事务 —— 从这里开始，所有操作都在“保护罩”内
await store.beginTransaction();

try {
    // 执行多条SQL语句
    await store.executeSql("INSERT INTO assets (name, price) VALUES (?, ?)", ["电脑", 5000]);
    await store.executeSql("UPDATE accounts SET balance = balance - 5000 WHERE id = ?", [1]);
    await store.executeSql("UPDATE accounts SET balance = balance + 5000 WHERE id = ?", [2]);
    
    // 全部成功 → 提交事务，数据真正写入
    await store.commit();
    console.log("事务提交成功，数据已持久化");
} catch (error) {
    // 任何一步出错 → 回滚事务，仿佛什么都没发生过
    await store.rollback();
    console.error("事务回滚，错误信息:", error);
}

🔑 核心口诀： begin -> 干活 -> 都成功就commit -> 有异常就rollback

    • 🧪 实训任务一：简单转账事务（15分钟）

  • 编写一个函数 transferAsset(assetId, fromAccount, toAccount)，实现资产从A库转移到B库。
  • 要求使用事务，确保扣减和增加同时成功或同时失败。
  • 完整代码示例： 
    async transferAsset(assetName: string, fromId: number, toId: number, amount: number) {
    let store = await this.getStore();
    await store.beginTransaction();
    try {
        // 扣减源账户
        await store.executeSql(
            "UPDATE accounts SET balance = balance - ? WHERE id = ?",
            [amount, fromId]
        );
        // 增加目标账户
        await store.executeSql(
            "UPDATE accounts SET balance = balance + ? WHERE id = ?",
            [amount, toId]
        );
        await store.commit();
        console.info("转移成功");
        return true;
    } catch (err) {
        await store.rollback();
        console.error("转移失败，已回滚:", err);
        return false;
    }
}
  • 测试： 调用 transferAsset("测试资产", 1, 2, 10)，观察两个账户余额变化。

    📋 场景： 从服务器或文件导入100条资产数据。如果第50条数据有问题（比如价格负数），我们希望整个批次都不导入，保持数据干净。

  • 准备模拟数据数组（可以在代码里写死用于测试）：

  • 编写批量导入函数，要求使用事务，并验证每条数据合法性（price > 0, quantity > 0）。
  • 如果任何一条不合法，整个批次回滚。
  • 参考代码框架： 
    async batchInsertWithTransaction(assets: Array<any>) {
    let store = await this.getStore();
    await store.beginTransaction();
    try {
        for (let item of assets) {
            // 数据校验
            if (item.price <= 0 || item.quantity <= 0) {
                throw new Error(`无效数据: ${item.name}, price=${item.price}`);
            }
            await store.executeSql(
                `INSERT INTO assets (name, category, price, quantity, create_time) 
                 VALUES (?, ?, ?, ?, ?)`,
                [item.name, item.category, item.price, item.quantity, new Date().toISOString()]
            );
        }
        await store.commit();
        console.info("全部导入成功");
    } catch (err) {
        await store.rollback();
        console.error("导入失败，已回滚:", err.message);
    }
}
  • 实操： 执行上述代码，观察错误数据导致全部回滚；修正错误数据后，再次执行，验证所有数据正确入库。

    • 🔥 进阶挑战： 使用 savepoint 实现“每10条提交一次”，既保证部分成功又不丢失大量数据。

    • 🧪 实训任务四：小组PK —— 事务性能对比（15分钟）

    ⚡ 挑战： 插入100条模拟数据，比较“逐条插入无事务” vs “批量插入使用事务”的耗时差异。

    规则： 每组编写测试代码，分别执行两种方式，记录耗时，看哪一组最快且数据正确。

    // 无事务方式（慢）
for (let i = 0; i < 100; i++) {
    await store.executeSql("INSERT INTO assets (name) VALUES (?)", [`资产${i}`]);
}

// 有事务方式（快）
await store.beginTransaction();
for (let i = 0; i < 100; i++) {
    await store.executeSql("INSERT INTO assets (name) VALUES (?)", [`资产${i}`]);
}
await store.commit();

    🏆 结论： 事务能极大提升批量写入性能（减少磁盘同步次数），同时保证一致性。
